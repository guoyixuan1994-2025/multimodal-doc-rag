# 3 分钟面试演示脚本

## 0:00 - 0:30：讲业务问题

“企业资料通常同时包含文字 PDF、扫描件、表格、公式和图片。传统文本 RAG 会丢失版面与视觉信息，所以我做了一个从多模态解析、混合检索、重排到可追溯问答的完整系统。”

## 0:30 - 1:20：展示多模态入库

1. 打开 Streamlit 控制台。
2. 上传 `sample_docs/scanned_ocr_sample.pdf`，选择 `ocr` 模式。
3. 展示异步解析任务进度，以及 MinerU 生成的结构化结果。
4. 再上传 `mixed_topology_image.png`，说明自然图片会保留 `image_caption`。

重点表达：不同文档类型会路由到不同解析路径，耗时任务可轮询、可取消。

## 1:20 - 2:20：展示检索与可追溯问答

1. 初始化样例知识库。
2. 提问：“为什么要用 BM25 和向量混合检索？”
3. 展示 query rewrite、BM25/向量混合召回、BGE reranker 分数和引用来源。
4. 提问一个资料中不存在的问题，展示系统拒答。

重点表达：系统不是只调用一次大模型，而是控制检索、重排、拒答和引用链路。

## 2:20 - 3:00：展示工程化

1. 打开 `/docs` 展示 FastAPI 接口。
2. 展示 Chroma 持久化、增量更新、同名文档覆盖和 collection 隔离。
3. 运行评估，展示 Hit、Precision@K、Recall@K、MRR 与 RAGAS 图表。
4. 最后打开 GitHub Actions，展示语法和 Docker Compose 自动校验。

## 面试官追问时可展开

- 为什么采用 BM25 + 向量混合召回，而不是只用向量检索？
- 如何避免多文档中同类图片或相似段落串答？
- 为什么自然图片问答不完全依赖通用文本 reranker？
- 如何控制 MinerU、embedding、reranker 和 LLM 的延迟与成本？
- 如果扩展到生产环境，如何增加对象存储、任务队列和租户隔离？
